That repetitive math was really bothering me.

It wasn‚Äôt clean.

Most of the repetition was between similar directions. For example, Oval.resizeLeft() had similarities with Header.resizeLeft(). This was because they both dealt with dragging the handle on the left side.

The other similarity was between the methods for the same shape. For example, Oval.resizeLeft() had similarities with the other Oval methods. This was because they all dealt with ovals. There was also some duplication between Rectangle, Header, and TextBlock because text blocks were rectangles.

I had an idea.

We could remove all duplication by grouping the code like this instead:

  },

  },

  },

    // 5 unique lines of math

  // 20 lines of code

  createHandle([top]),
  createHandle([left]),
  createHandle([right]),
  createHandle([bottom]),
];

  createHandle([left]),
  createHandle([right]),

let Header = createBox(Shapes.Rectangle, twoSides);
let TextBox = createBox(Shapes.Rectangle, fourCorners);

The code is half the total size, and the duplication is gone completely! So clean. If we want to change the behavior for a particular direction or a shape, we could do it in a single place instead of updating methods all over the place.

Obsessing with ‚Äúclean code‚Äù and removing duplication is a phase many of us go through. You can‚Äôt automate removing duplication, but it does get easier with practice. You can usually tell whether there‚Äôs less or more of it after every change. It‚Äôs as powerful as any sort of self-deception.

Once we learn how to create abstractions, it is tempting to get high on that ability, and pull abstractions out of thin air whenever we see repetitive code. I see now that my ‚Äúrefactoring‚Äù was a disaster in two ways:

  • Secondly, nothing is free. My code traded the ability to change requirements for reduced duplication, and it was not a good trade. My abstraction would have to become several times more convoluted to afford that, whereas with the original ‚Äúmessy‚Äù version such changes stayed easy as cake.

I sure didn‚Äôt think deeply about any of those things. I thought a lot about how the code looked ‚Äî but not about how it evolved with a team of squishy humans.

Coding is a journey. Think how far you came from your first line of code to where you are now. I reckon it was a joy to see for the first time how extracting a function or refactoring a class can make convoluted code simple. If you find pride in your craft, it is tempting to pursue cleanliness in code. Do it for a while.

But don‚Äôt stop there. Don‚Äôt be a clean code zealot. Clean code is not a goal. It‚Äôs an attempt to make some sense out of the immense complexity of systems we‚Äôre dealing with.
